# Night Shift Survival ロードマップ

## 1. 前提
このロードマップは `仕様書.md` に記載した 2026-03-24 時点の現行仕様を基準にしている。  
現在の到達点は「単一 Night を短時間で遊べるブラウザ試作」であり、今後は以下の順で完成度を高める。

## 2. 開発方針
- まずは現行ループの読み合いを深くする
- 次にコンテンツ量と進行構造を増やす
- その後に演出品質と世界観の独自性を高める
- 常に変更範囲を最小化し、仕様と実装の乖離を抑える

## 3. フェーズ別計画

### Phase 1: 現行試作のコア改善
目的:
現状の「遊べる試作」を、判断ゲームとして成立度の高い試作へ引き上げる。

主要項目:
- Threat Readout を難易度別またはモード別に整理し、常時完全表示から脱却する
- 敵ごとの対処法を明確化し、`Rust` / `Moth` / `Grin` に distinct な読み合いを与える
- カメラを索敵 UI から判断装置へ寄せ、見る意味と見すぎるリスクを増やす
- 線形な AI 上昇だけに頼らず、時間帯ごとの緊張の波を設計する

完了条件:
- Threat Readout を見続けるだけで最適化できない
- 敵ごとにプレイヤーの対応が変わる
- カメラ確認に明確な利得とリスクがある

### Phase 2: 保守性と実装基盤の整理
目的:
Night 制や難易度追加に耐える構造へ整理する。

主要項目:
- `game.js` の責務を状態管理、UI、描画、音、データに分割する
- 敵定義、カメラ定義、文言定義をデータ駆動化する
- 調整対象の数値をコード直書きからまとめ、再調整しやすくする
- 簡易の検証手順を整備し、仕様更新時の確認コストを下げる

完了条件:
- 新敵や新カメラ追加時の修正箇所が限定される
- 主要数値がまとまった定義から変更できる
- 仕様変更時の確認手順が文書化されている

### Phase 3: 進行要素の追加
目的:
単発試作から、継続して遊べる構造へ拡張する。

主要項目:
- Night 1 以降の段階解放を実装する
- 難易度選択を導入し、ヒント量や補助表示を調整可能にする
- `localStorage` を用いた設定保存と進行保存を導入する
- 進行に応じて敵構成や初期条件を変え、学習曲線を設計する

完了条件:
- 複数 Night を順に攻略できる
- 難易度差が UI とゲーム挙動の両方に反映される
- 言語や音声設定、進行状況が保存される

### Phase 4: 演出と独自性の強化
目的:
FNaF 風試作から、作品としての独自性を持つホラーゲームへ寄せる。

主要項目:
- 施設設定、勤務理由、敵の存在理由など世界観を定義する
- UI をより監視室らしい質感へ寄せ、故障感や緊急感を強化する
- 敵ごとの固有ビジュアル、専用音、前兆演出を追加する
- 開始前メッセージ、異常ログ、クリア後演出を整備する

完了条件:
- 画面、音、文言から独自の舞台設定が伝わる
- 敵を見ただけで種類や危険性が判別できる
- 勝利と敗北の体験差が現在より明確になる

## 4. 優先順位
直近で着手すべき順序は以下とする。

1. Phase 1 の Threat Readout 整理
2. Phase 1 の敵ごとの対処法強化
3. Phase 1 のカメラ役割強化
4. Phase 2 のデータ整理と責務分離
5. Phase 3 の Night 制と保存
6. Phase 4 の演出強化と世界観整備

## 5. 直近の実装候補
次の 1 スプリントで扱いやすい候補は以下。

- Threat Readout に `Prototype` / `Standard` の表示モードを追加する
- `Moth` の活性化を UI と音でより把握しやすくする
- カメラごとに前兆やノイズ差分を追加する
- 敵定義を 1 か所に集約し、数値調整をしやすくする
