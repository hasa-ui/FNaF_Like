# Night Shift Survival 仕様書（更新版）

## 1. 概要

### 1-1. 作品概要
本作は、監視カメラ・左右ドア・電力管理を用いて、深夜勤務中の警備室を防衛し、午前 6 AM まで生存することを目的としたブラウザ用ホラーゲームである。

現時点では単発ナイト型の試作版だが、すでに以下の基本要素は実装済みである。

- タイトル画面
- 操作説明画面
- オフィス / カメラ画面の切替
- 左右ドアの開閉
- 電力管理
- 敵の接近と敗北判定
- 勝利 / 敗北表示
- 簡易音声
- 簡易ジャンプスケア
- 日本語 / 英語切替

今後は、単なる「FNaF風の見た目をした管理ゲーム」に留まらず、**敵ごとに対処法が異なる監視ホラー**としての完成度を高めることを目標とする。

### 1-2. ジャンル
- ホラー
- サバイバル
- リソース管理
- 監視・防衛ゲーム

### 1-3. 対応環境
- デスクトップブラウザ
- スマホブラウザは簡易対応段階

現状はレスポンシブ対応が入っているが、情報量と操作量の都合上、主対象はデスクトップブラウザとする。

### 1-4. 実装構成
現行版は以下の 3 ファイル構成で動作する。

- `index.html`
- `style.css`
- `game.js`

依存ライブラリなしの静的フロントエンド構成であり、ローカル実行可能である。

## 2. ゲームコンセプト

### 2-1. コア体験
プレイヤーは、限られた電力の中で以下を判断し続ける。

- カメラを確認する
- 左右ドアを閉じる
- 何もせず節電する

本作で重要なのは反射神経だけではなく、**情報取得とリスク管理のバランス**である。

### 2-2. 体験設計の核
本作の緊張感は、以下の関係から生まれる。

- 情報を得るほど安全になるが、電力を消費する
- 守りを固めるほど即死を防げるが、電力を消費する
- 節電しすぎると敵接近への対応が遅れる
- 時間経過により敵が強化され、同じ判断でも通用しなくなる

### 2-3. 作品の目標
本作は今後、以下を満たすゲーム体験を目指す。

- 敵ごとに異なる対応を求めること
- カメラが単なる位置確認装置ではなく、判断材料になること
- UI だけでなく、映像・音・演出も含めて不安を作ること
- 難易度が単なる数値増加ではなく、緊張の波として設計されること

## 3. 現在の実装仕様

### 3-1. 画面構成
現行 UI は以下で構成される。

#### 上部バー
- タイトル表示
- 音声オン / オフ切替
- 言語切替
- 現在時刻表示

#### メイン画面
- オフィス画面
- カメラ画面
- タイトル画面オーバーレイ
- How To Play オーバーレイ
- カメラ切替演出レイヤー
- ジャンプスケア演出レイヤー

#### 下部 HUD
- 電力表示
- ドア状態表示
- ステータス表示

#### 右カラム
- 操作ボタン
- カメラマップ
- Threat Readout
- 勝敗表示パネル

### 3-2. ゲーム進行
- 開始時刻は `12 AM`
- 1 秒ごとに内部時間が 1 進行する
- 10 単位でゲーム内 1 時間進行した扱いとなる
- `minutesElapsed >= 60` で `6 AM` 到達、勝利となる

つまり、現行版は **約 60 秒で 1 プレイが完結する短時間設計** である。

### 3-3. ゲーム状態
ゲーム状態は以下を持つ。

- `title`
- `howto`
- `playing`
- `win`
- `lose`

これにより、タイトル・説明・プレイ中・勝敗後の画面を切り替えている。

### 3-4. プレイヤー操作
プレイヤーが行える操作は以下。

- 左ドアの開閉
- 右ドアの開閉
- モニターの上げ下げ
- カメラ切替
- ゲーム開始
- How To Play 表示
- タイトルに戻る
- リスタート
- 音声オン / オフ切替
- 日本語 / 英語切替

### 3-5. 入力制御
現行版では以下の入力制御が入っている。

- カメラ切替演出中は入力を一時ロックする
- モニター展開中は左右ドア操作を禁止する
- タイトル / How To 状態ではプレイ用入力を無効にする
- 勝利 / 敗北後はプレイ操作を無効にする

これにより、操作の暴発や演出中の不整合を防いでいる。

### 3-6. カメラ
現在監視可能なカメラは以下。

- `stage`
- `hallL`
- `hallR`
- `storage`
- `office`

表示名は言語設定に応じて切り替わる。

#### カメラの現状の役割
- 敵位置の確認
- 雰囲気演出
- 状況把握

ただし現状では、**実質的には位置確認手段としての役割が強い**。

### 3-7. リソース管理
#### 電力
- 初期値は `100%`
- 時間経過で減少する
- 使用設備に応じて減少量が増える

#### 消費ルール
基本使用量:
- 常時 `1`

追加使用量:
- 左ドア閉鎖中 `+1`
- 右ドア閉鎖中 `+1`
- カメラ使用中 `+1`

実際の減少量:
- 1 秒ごとに `0.42 × 使用量`

#### 電力切れ時
電力が 0 以下になると、以下が発生する。

- 左右ドアが強制的に開く
- カメラが強制的に閉じる
- 停電ステータスに切り替わる
- 危険位置に敵がいれば一定時間後に敗北する
- 停電用サウンドが一度だけ鳴る

### 3-8. 敵キャラクター
現行版では以下の 3 体が存在する。

- Rust
- Moth
- Grin

#### 役割差
現時点での差別化は主に以下。

- 移動ルートの違い
- 移動確率の違い
- 入口到達時の猶予時間の違い
- Moth のみ、モニター使用に反応して活性化する

#### 現行ルート
- Rust: `stage -> office -> hallL`
- Moth: `stage -> storage -> hallR`
- Grin: `storage -> office -> hallL -> hallR`

厳密には、各敵は一定確率でルート上を前進し、`office` 到達後は左右入口側へ遷移する。

### 3-9. 敵AI
#### 初期 AI 値
- Rust: `0.24`
- Moth: `0.18`
- Grin: `0.14`

#### 上昇量
3 秒ごとに以下のように増加する。

- Rust: `+0.015`
- Moth: `+0.013`
- Grin: `+0.018`

#### 上限
- Rust: `0.92`
- Moth: `0.90`
- Grin: `0.94`

#### 個別特性
- Rust  
  左入口側で危険性が高い。左到達時は脅威として分かりやすい。
- Moth  
  モニターの上げ下げや連続使用に反応し、`mothAggro` によって移動しやすくなる。
- Grin  
  左右に跨るルートを持ち、追跡しづらい。Threat Readout 上でも不安定表示になる場合がある。

### 3-10. 敗北条件
現在の主な敗北条件は以下。

#### 左側侵入
- Rust または Grin が `hallL`
- 左ドアが開いている
- 一定猶予経過後に敗北

#### 右側侵入
- Moth または Grin が `hallR`
- 右ドアが開いている
- 一定猶予経過後に敗北

#### 停電中の襲撃
- 電力が 0
- 敵が `office / hallL / hallR` に存在
- 一定時間後に敗北

#### 敗北演出
- ジャンプスケア音
- ジャンプスケア画面
- その後に敗北パネル表示

### 3-11. 勝利条件
- ゲーム内時刻が `6 AM` に到達すること

勝利時には勝利パネルが表示され、勝利音が鳴る。

### 3-12. HUD と補助表示
現行 HUD には以下が含まれる。

- 現在時刻
- 電力残量
- 使用量
- 左右ドア状態
- ステータス文
- ヒント文
- カメラ一覧
- Threat Readout

#### Threat Readout の位置づけ
Threat Readout は現状、**試作・調整・練習用の補助情報**である。現行実装では敵名や位置情報をかなり強く提示しており、本番仕様としては情報量が多すぎる。

### 3-13. 多言語対応
現行版では以下の言語切替が可能である。

- 日本語
- 英語

文言は内部辞書で管理され、以下の内容が言語切替対象となる。

- タイトル / メニュー文言
- 操作説明
- HUD ラベル
- ステータス文
- ボタン文言
- カメラ名
- カメラ説明
- 勝敗文言

### 3-14. オーディオ
現行版では Web Audio API による簡易音を実装済みである。

#### 実装済みの音
- モニター上げ
- モニター下げ
- ドア操作
- 脅威接近
- 停電
- ジャンプスケア
- 勝利
- 簡易環境音

#### 現状の限界
- 左右定位がない
- 敵ごとの差が弱い
- 実音素材ではなく簡易波形中心
- 雰囲気強化よりも「最低限音が鳴る」段階に近い

### 3-15. 演出
現状の演出要素は以下。

- 暗いオフィス画面
- カメラ風ノイズ
- 走査線
- カメラ切替アニメーション
- 敵接近時の表示
- ジャンプスケア演出
- 敗北オーバーレイ
- メニューオーバーレイ

#### 未実装または弱い点
- 停電中の専用ビジュアル強化
- 敵ごとの専用演出差
- 勝利演出の強化
- UI 劣化表現
- 高度なホラー音響演出

## 4. 現在仕様の評価

### 4-1. 良い点
- 1 プレイが短く、試行錯誤しやすい
- タイトル、説明、ゲーム本編、勝敗まで一通り成立している
- 音声、言語切替、演出レイヤーが追加され、作品感が増した
- HTML / CSS / JS のみで構成され、ローカルで動く
- Moth のように、プレイヤー行動に反応する敵の芽がある
- 入力ロックにより、演出と操作の整合性が改善されている

### 4-2. 現在の問題点
- 敵ごとの個性がまだ弱い
- カメラが位置確認 UI に寄りすぎている
- Threat Readout が強すぎて不確実性を壊している
- 難易度上昇が線形で、山場ができにくい
- 音は入ったが、判断材料としてはまだ弱い
- `game.js` に責務が集中しており、拡張時に崩れやすい
- 敵データが分散しており、追加や調整の保守性が低い
- UI がまだ「監視室」より「綺麗なダッシュボード」寄り

### 4-3. このまま機能追加した場合のリスク
このまま単純に敵数・カメラ数・演出量だけを増やすと、**見た目は豪華だが読み合いの浅いゲーム**になりやすい。

特に以下が曖昧なままだと限界が来る。

- 敵ごとに何が怖いのか
- プレイヤーが何を根拠に判断するゲームなのか
- カメラを見る意味は何か
- 情報の信頼性をどの程度揺らすか
- どこで緊張のピークを作るか

## 5. 更新後の設計方針

### 5-1. 基本方針
本作は今後、単なる「左右ドア管理ゲーム」ではなく、**敵ごとに異なる対応を要求する監視ホラー**へ進化させる。

そのための中核方針は以下。

- 情報の価値を高める
- 敵ごとの対処法を分ける
- 不確実性を残す
- 数値だけではない緊張の波を作る
- UI、演出、音を含めて監視室体験を強化する

### 5-2. 優先原則
追加機能の優先順位は「量」ではなく「体験への寄与」で決める。

優先すべき順は以下。

1. Threat Readout の本番仕様化
2. 敵ごとの対処法の明確化
3. カメラの役割強化
4. コード構造の整理
5. 音と演出の質向上
6. Night 制と難易度設計

## 6. 今後の仕様更新案

### 6-1. 優先度A: 最優先で改修すべき仕様

#### A-1. Threat Readout の本番仕様化
現状:
- 敵位置が直接的に見えすぎる
- プレイヤーが映像より HUD を読む状態になりやすい

変更方針:
- 開発用 / 練習用では残してよい
- 通常難易度では非表示または曖昧表示にする
- 敵名と位置の完全表示を避ける

難易度別例:
- Easy: 位置の概略表示あり
- Normal: 危険方向ヒントのみ
- Hard: 表示なし

#### A-2. 敵ごとの対処法の明確化
現状:
- 主な差は数値とルート差

変更方針:
- Rust: 接近は分かりやすいが、入口到達後の猶予が短い
- Moth: モニター使用に反応する
- Grin: 追跡しづらく、位置が読みづらい
- 新敵追加時は「対処法の差」を必須条件とする

要求仕様:
- 各敵は最低 1 つ、他敵と異なる対処原理を持つこと
- 全敵に対して同じ行動が最適解にならないこと

#### A-3. カメラの役割強化
現状:
- 実質的に位置確認装置

変更方針:
- カメラごとに役割差を持たせる
- 一部敵は「見ること」が対処になる
- 一部敵は「見すぎること」が逆効果になる
- 兆候を観察する価値を増やす

例:
- 特定カメラでしか前兆が見えない
- ノイズの中にヒントが混ざる
- 一部カメラの情報精度が低い
- 映像にフェイク要素が混ざる

#### A-4. 難易度上昇の再設計
現状:
- 3 秒ごとの線形強化

変更方針:
- 時間帯ごとに敵挙動を変える
- 静かな時間帯と崩れる時間帯を分ける
- 5 AM 付近でピークを作る
- 単純な AI 上昇だけに依存しない

#### A-5. コード構造の整理
現状:
- `game.js` に文言、音、描画、ロジック、状態管理が集中している

変更方針:
- 責務分離を進める
- 敵データを一元化する
- 今後の Night 制や難易度対応に備える

### 6-2. 優先度B: 深さを増す仕様

#### B-1. Night 制
目的:
- 継続的な達成感と進行要素を追加する

仕様案:
- Night 1, Night 2, Night 3... を段階解放
- 夜ごとに敵構成、AI初期値、ヒント量を変える
- 新ルールは段階的に開放する

#### B-2. 難易度選択

調整対象:

- AI 初期値

- 時間進行速度
- カメラ情報精度

#### B-3. オーディオ強化
現状:
- 簡易効果音中心

変更方針:
- 左右定位を追加する
- 敵ごとの固有音を持たせる
- 接近度で音を変える
- 停電中の無音や低音で緊張感を作る

#### B-4. UI の監視室化
現状:
- 見やすいが、少し綺麗すぎる

変更方針:
- 角丸を減らす
- 故障感、古さ、不穏さを強める

## 7. 非機能要件

### 7-1. パフォーマンス
- 一般的なブラウザで軽量動作すること
- 演出追加後も過剰に重くならないこと
- DOM 更新による大きなフレーム落ちを避けること

### 7-2. 可読性
- ロジック、UI、音、データを分離すること
- 文言辞書を別管理できる構造を目指すこと
- 敵追加時に修正箇所が散らばらないこと

### 7-3. 拡張性
- 敵追加時に大規模改修が不要なこと
- カメラ追加時に定義配列レベルの修正で済むこと
- Night や難易度をデータで管理できること

### 7-4. セーブ
将来的には以下を保存対象とする。

- 解放済み Night
- 難易度設定
- 言語設定
- 音量 / 音声設定
- 既読ログ
- 練習モード設定

保存方式は `localStorage` を第一候補とする。

## 10. 総括
現行版の Night Shift Survival は、すでに「遊べる試作」から「作品らしさを持った試作」へ進んでいる。  
多言語化、簡易音声、カメラ切替演出、ジャンプスケア、入力ロックなどが入り、前段階より完成度は確実に上がっている。

ただし、今後の課題は演出量よりも、**ゲーム設計の深さとコード構造の耐久性**にある。

特に優先すべきなのは以下の 5 点である。

1. Threat Readout の整理  
2. 敵ごとの対処法の明確化  
3. カメラを「ただの索敵」から脱却させること  
4. `game.js` の責務分離  
5. 敵定義のデータ駆動化  

現時点で最も価値が高い次の一手は、**「Threat Readout の本番仕様化」「敵個性の強化」「カメラの役割強化」** である。
