# Night Shift Survival 仕様書（更新版）

## 1. 概要

### 1-1. 作品概要
本作は、監視カメラ・左右ドア・電力管理を用いて、深夜勤務中の警備室を防衛し、午前 6 AM まで生存することを目的としたブラウザ用ホラーゲームである。現時点では短時間で遊べる単発ナイト型の試作版であり、今後は敵の個性、情報取得の意味、演出、進行要素を強化していく。現行実装ではタイトル画面、操作説明画面、カメラ監視、左右ドア、防衛失敗・勝利表示までが一通り動作する。

### 1-2. ジャンル
- ホラー
- サバイバル
- リソース管理
- 監視・防衛ゲーム

### 1-3. 対応環境
- デスクトップブラウザ
- スマホブラウザは簡易対応段階

現行 CSS では 980px 以下で 1 カラム化するレスポンシブ対応が入っているが、操作量と情報量の都合上、現時点ではデスクトップ前提の調整が望ましい。

### 1-4. 現在の実装構成
現行版は以下の 3 ファイル構成で動作する。

- `index.html`
- `style.css`
- `game.js`

依存ライブラリなしの静的フロントエンド構成であり、ローカル実行可能である。

## 2. ゲームコンセプト

### 2-1. コア体験
プレイヤーは、限られた電力の中で以下を判断し続ける。

- カメラを確認する
- 左右ドアを閉じる
- 何もせず節電する

重要なのは反射神経だけではなく、情報取得とリスク管理のバランスである。

### 2-2. 体験設計の核
本作の中核は、以下の緊張関係にある。

- 情報を得るほど安全になるが、電力を消費する
- 守りを固めるほど即死を防げるが、電力を消費する
- 節電しすぎると敵接近への対応が遅れる
- 時間経過により敵が強化され、同じ操作でも通用しなくなる

### 2-3. 現状の課題
現行版はゲームループ自体は成立しているが、まだ「監視ホラーとしての深さ」は不足している。主な課題は以下。

- 敵ごとの対処法の差がまだ弱い
- カメラが「位置確認手段」に寄りすぎている
- Threat Readout により不確実性が薄れている
- 難易度上昇が線形で、山場が弱い
- UI がやや綺麗すぎて、監視室ホラーとしての不穏さが足りない

## 3. 現在の実装仕様

### 3-1. 画面構成
現行 UI は大きく以下で構成される。

- 上部情報バー
  - タイトル
  - 現在時刻
- メイン画面
  - オフィス表示
  - カメラ表示
  - タイトル／How To オーバーレイ
- 下部 HUD
  - Power
  - Doors
  - Status
- 右カラム
  - Controls
  - Camera Map
  - Threat Readout
  - End Panel

タイトル画面、How To Play、スタート導線はすでに実装済みである。

### 3-2. ゲーム進行
- 開始時刻は `12 AM`
- 1 秒ごとに内部時間が 1 進行する
- 10 単位でゲーム内 1 時間進行した扱いとなる
- `minutesElapsed >= 60` で `6 AM` 到達、勝利となる

つまり、1 プレイは約 60 秒前後で完結する短時間設計である。

### 3-3. ゲーム状態
ゲーム状態は以下を持つ。

- `title`
- `howto`
- `playing`
- `win`
- `lose`

これにより、タイトル・説明・プレイ中・勝敗後を単一状態管理で切り替えている。

### 3-4. プレイヤー操作
プレイヤーが可能な操作は以下。

- 左ドア開閉
- 右ドア開閉
- カメラタブレットの上げ下げ
- カメラ切り替え
- タイトルへ戻る
- タイトル画面から開始
- How To 画面への移動
- How To 画面から開始

#### 現在の監視可能カメラ
- `STAGE`
- `HALL L`
- `HALL R`
- `STORAGE`
- `OFFICE OUT`

カメラ一覧は `CAMS` 配列として定義されている。

### 3-5. リソース管理
#### 電力
- 初期値は `100%`
- 時間経過で減少
- 使用設備に応じて消費量増加

#### 現行の消費ルール
基本使用量:
- 常時 `1`

追加使用量:
- 左ドア閉鎖中 `+1`
- 右ドア閉鎖中 `+1`
- カメラ使用中 `+1`

実際の減少量:
- 1 秒ごとに `0.42 × 使用量`

#### 電力切れ時
電力が 0 以下になると以下が発生する。

- 左右ドアが強制開放
- カメラが閉じる
- ステータスが停電状態になる
- 敵が危険圏にいれば、一定時間後に敗北する

この処理は秒ごとのループ内で行われる。

### 3-6. 敵キャラクター
現行版では 3 体の敵が存在する。

- Rust
- Moth
- Grin

#### 現在の役割差
現時点での差別化は主に以下。

- 移動ルートの違い
- 移動確率の違い
- 入口到達時の猶予時間の違い
- Moth のみカメラ使用で警戒値が上がる

#### 現行ルート
- Rust: `stage -> office -> hallL`
- Moth: `stage -> storage -> hallR`
- Grin: `storage -> office -> hallL -> hallR`

厳密には、ルート配列に沿って一定確率で前進し、`office` 到達後は左右入口側へ戻る処理が入っている。

### 3-7. 敵AI
#### 初期 AI 値
- Rust: `0.24`
- Moth: `0.18`
- Grin: `0.14`

#### 上昇量
3 秒ごとに以下のように増加する。

- Rust: `+0.015`
- Moth: `+0.013`
- Grin: `+0.018`

#### 上限
- Rust: `0.92`
- Moth: `0.90`
- Grin: `0.94`

#### 個別特性
- Rust  
  左側到達時に危険度が高い。`hallL` 付近で補正が入る。
- Moth  
  カメラを開いている間、さらに警戒値 `mothAggro` が上がる。
- Grin  
  左右に跨るルートを持ち、位置の追跡がやや読みづらい。

ただし現状では、敵ごとの個性はまだ「対処法の違い」というほど強くはなく、主に数値差と移動パターン差に留まっている。

### 3-8. 敗北条件
現在の主な敗北条件は以下。

#### 左側侵入
- Rust または Grin が `hallL`
- 左ドアが開いている
- 敵ごとの侵入猶予時間経過で敗北

#### 右側侵入
- Moth または Grin が `hallR`
- 右ドアが開いている
- 敵ごとの侵入猶予時間経過で敗北

#### 停電中の襲撃
- 電力 0
- 敵が `office / hallL / hallR` のいずれかに存在
- 一定時間後に敗北

敗北後は End Panel と画面内オーバーレイを表示する。

### 3-9. 勝利条件
- ゲーム内時刻が `6 AM` に到達すること

勝利時には End Panel に `Shift Complete` を表示する。

### 3-10. HUD と補助表示
現行 HUD は以下を表示する。

- 現在時刻
- 電力残量
- 使用量
- 左右ドア状態
- ステータス文
- ヒント文
- カメラ一覧
- Threat Readout

#### Threat Readout の位置づけ
Threat Readout は現状、試作・調整・練習用の可視化機能であり、本番仕様としては強すぎる。通常難易度以上では非表示または曖昧表示へ移行することを前提とする。

### 3-11. 演出
現状の演出要素は以下。

- 暗いオフィス画面
- カメラ風ノイズ
- 走査線表現
- 敵接近時の簡易表示
- 敗北オーバーレイ
- タイトル／How To オーバーレイ

一方で、以下は未実装である。

- 音全般
- 本格的なジャンプスケア
- 停電時の専用演出強化
- 敵ごとの演出差
- 勝利時の特別演出

## 4. 現在仕様の評価

### 4-1. 良い点
- 1 プレイが短く、試行錯誤しやすい
- タイトル、説明、ゲーム本編、勝敗まで一周の体験が成立している
- HTML / CSS / JS のみで構成され、改造しやすい
- 「見る」「閉じる」「待つ」の基本判断が明快
- Moth のような、行動に反応する敵の芽がすでにある

### 4-2. 現在の問題点
- 敵ごとの個性がまだ弱い
- カメラが位置確認 UI に寄りすぎている
- Threat Readout により緊張の核である不確実性が薄い
- 難易度上昇が線形で、山場ができにくい
- UI が綺麗に整いすぎており、監視室ホラーの不穏さが弱い
- 音がないため、恐怖演出の大きな柱が欠けている
- グローバル状態中心で、今後の拡張時に保守性が落ちやすい

### 4-3. このまま機能追加した場合のリスク
このまま単純に敵・カメラ・数値だけを増やすと、「FNaF 風の見た目をした管理ゲーム」に留まりやすい。今後は機能追加よりも先に、以下を明確化する必要がある。

- 敵ごとに何が怖いのか
- 何を見て判断するゲームなのか
- 情報がどれだけ信頼できるのか
- プレイヤーにどんな緊張の波を作るのか

## 5. 更新後の設計方針

### 5-1. 基本方針
本作は今後、単なる「左右ドア管理」から一歩進め、**敵ごとに異なる対処法を要求する監視ホラー**へ進化させる。その際、以下を中核方針とする。

- 情報の価値を高める
- 敵ごとの対処法を分ける
- 不確実性を残す
- 線形ではない恐怖の山場を作る
- UI・演出・音を含めて監視室体験を強める

### 5-2. 仕様追加の優先原則
優先順位は「量」ではなく「体験への寄与」で決める。  
そのため、カメラ追加数や敵数の増加よりも、まず以下を優先する。

1. 敵個性の明確化
2. Threat Readout の本番仕様化
3. カメラの価値強化
4. 演出と音の追加
5. Night 制や難易度の導入

## 6. 今後の仕様更新案

### 6-1. 優先度A: 最優先で改修すべき仕様

#### A-1. Threat Readout の通常仕様変更
現状:
- 敵位置が直接的に見えすぎる

変更方針:
- 開発用／練習用では残してよい
- 通常難易度では非表示または曖昧表示にする
- 敵名と位置の完全表示は避ける

難易度別例:
- Easy: 位置の概略表示あり
- Normal: 危険側ヒントのみ
- Hard: 表示なし

#### A-2. 敵ごとの対処法の明確化
現状:
- 主な差は数値とルート差

変更方針:
- Rust: 到達後の猶予が短く、入口対処重視
- Moth: カメラ確認回数に反応する監視依存型
- Grin: 位置把握が困難でフェイント気味
- 追加敵を入れる場合も「別の対処」を必須条件とする

要求仕様:
- 各敵は最低 1 つ、他敵と異なる対処原理を持つこと
- 「全敵に対して同じ対応」が最適解にならないこと

#### A-3. カメラの役割強化
現状:
- 実質的に位置確認装置

変更方針:
- カメラごとに意味を持たせる
- 監視しないと見えない前兆を増やす
- 一部敵は「見ること」自体が対処になる、または逆効果になる

例:
- 特定カメラでしか兆候がわからない
- ノイズの中にヒントが混ざる
- 一部カメラは情報精度が低い
- 長時間監視で別の敵が活発化する

#### A-4. 敗北演出強化
現状:
- オーバーレイ中心で淡泊

変更方針:
- 敗北直前に演出ピークを作る
- フラッシュ、ノイズ増加、画面ぶれ等を追加する
- 将来的に音と組み合わせて印象を強める

#### A-5. オーディオ実装
必須音:
- 環境音
- カメラ開閉音
- ドア開閉音
- 停電音
- 敵接近音
- 勝利音
- 敗北音／ジャンプスケア音

設計原則:
- 音は飾りではなく判断材料として使う
- 左右の危険を定位や種類で示せるようにする

### 6-2. 優先度B: 深さを増す仕様

#### B-1. Night 制
目的:
- 継続的な成長と達成感の追加

仕様案:
- Night 1, Night 2, Night 3... を解放制にする
- 夜ごとに AI 初期値、敵構成、ヒント量を変える
- 新しい敵特性は段階的に解放する

#### B-2. 難易度選択
候補:
- Casual
- Standard
- Hard
- Custom Night

調整対象:
- 初期電力
- AI 初期値
- Threat Readout の表示
- ヒント頻度
- 時間進行速度
- カメラ情報の精度

#### B-3. 緊張の山場設計
現状:
- AI 上昇が線形

変更方針:
- 時間帯ごとに敵の挙動を変える
- 4 AM 以降だけ特殊行動を解禁する
- 静かな区間と急激に崩れる区間を意図的に作る

#### B-4. 情報の不確実性
高難度向け仕様として以下を検討する。

- カメラ映像遅延
- 偽信号
- ノイズによる見落とし
- 敵のフェイント
- 状況に応じて HUD の信頼性低下

ただし初期段階で入れすぎると理不尽になるため、段階解放とする。

### 6-3. 優先度C: 作品性を作る仕様

#### C-1. 世界観の確立
明確化すべき項目:
- 施設の種類
- プレイヤーの立場
- 敵が活動する理由
- なぜ 6 AM まで耐える必要があるのか

#### C-2. 固有ビジュアルの強化
方針:
- 既存ホラー監視ゲームの印象から離す
- 敵・施設・UI を独自方向へ再設計する

候補:
- 研究施設
- 倉庫
- 展示館
- 保管区域
- 封鎖施設

#### C-3. 導入演出・記録物
追加候補:
- 夜開始前メッセージ
- 勤務メモ
- 館内アナウンス
- 異常記録ログ
- クリア報告文

#### C-4. エンディング差分
候補:
- 通常クリア
- 全夜クリア
- ノーミス系条件
- 隠し条件達成

## 7. 非機能要件

### 7-1. パフォーマンス
- 一般的なブラウザで軽量に動作すること
- 音・画像・演出追加後も過剰に重くならないこと
- DOM 操作の増加によってフレーム落ちしないこと

### 7-2. 可読性
- ロジック・UI・音・データを責務分離すること
- 敵追加時に分岐の散乱を最小限にすること
- 状態変化の責任箇所を明確にすること

### 7-3. 拡張性
- 敵追加時に複数箇所のベタ修正を減らす
- カメラ追加時は定義配列の修正中心で済む構造にする
- 難易度や Night 情報をデータで管理できるようにする

### 7-4. セーブ
将来的に以下を保存対象とする。

- 解放済み Night
- 難易度設定
- 既読ログ
- 音量設定
- 練習モード設定

保存方式は `localStorage` を第一候補とする。

## 8. 実装構成の更新方針

### 8-1. 現行の問題
現行版では `state` とタイマー群がグローバル寄りに管理され、敵情報も複数の定義に分散している。このため、試作としては十分だが、拡張には不利である。

### 8-2. 推奨ファイル構成
初期:
- `index.html`
- `style.css`
- `game.js`

拡張後の推奨:
- `game.js` : 起動制御
- `logic.js` : ゲーム進行、勝敗判定
- `ui.js` : DOM 更新、描画
- `audio.js` : 音管理
- `data.js` : 敵、カメラ、Night、難易度定義
- `save.js` : セーブ管理

### 8-3. データ駆動化対象
- 敵ごとの AI 値
- 敵ごとのルート
- 敵ごとの対処タイプ
- カメラ一覧
- Night ごとのパラメータ
- 難易度設定
- 表示制御設定

## 10. 総括
現行版の Night Shift Survival は、すでに「遊べる試作」として成立している。タイトル、説明、カメラ、ドア、電力、勝敗という最小限のゲームループは揃っており、土台自体は悪くない。

ただし、このままでは「それっぽい雰囲気の管理ゲーム」に留まりやすい。完成度を上げるには、単純な機能追加より先に、

- 敵ごとの対処法の違い
- カメラを見る意味
- 情報の不確実性
- 演出と音による緊張感
- 継続プレイを支える Night 構造

を優先して設計する必要がある。

現時点で最も価値が高い次の一手は、**「Threat Readout の整理」「敵個性の明確化」「カメラの役割強化」** の 3 点である。
